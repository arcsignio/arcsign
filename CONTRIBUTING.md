# Contributing to ArcSign

Thank you for considering contributing! Before opening a PR, please read
this entire document — especially the **"What we do NOT accept"** section.

> ArcSign is a one-maintainer project. To keep maintenance sustainable,
> we have strict acceptance criteria. This is not personal.

## TL;DR

1. Open an issue **first** for any non-trivial change (>50 lines).
2. All commits must be signed off (`git commit -s`).
3. PRs touching `internal/wallet/constants.go`, `internal/security/`, or
   `contracts/` require pre-discussion and extended review.
4. Tests required for non-trivial changes.

## What we DO accept

- ✅ Typo fixes, documentation improvements, translations.
- ✅ Bug fixes (with regression test) for confirmed issues with an open issue.
- ✅ New chain adapters following the existing `src/chainadapter/` pattern.
- ✅ Test coverage improvements.
- ✅ Performance improvements with benchmarks showing the improvement.
- ✅ Security hardening (please prefer email disclosure first — see
  [`SECURITY.md`](SECURITY.md)).

## What we do NOT accept

- ❌ Large rewrites without prior discussion in an issue.
- ❌ "Add support for X" without an open issue.
- ❌ Refactoring PRs that don't fix a concrete bug.
- ❌ Style / format changes that deviate from existing patterns.
- ❌ PRs without tests for non-trivial changes.
- ❌ Changes to compiled constants in
  [`internal/wallet/constants.go`](internal/wallet/constants.go)
  (these are official addresses — changing them creates a fork, not a
  contribution; see [`OFFICIAL_ADDRESSES.md`](OFFICIAL_ADDRESSES.md)).
- ❌ Dependencies added without justification of necessity.
- ❌ Code generated by AI without review and understanding by the submitter.

### Prerequisites

- Go 1.21+
- Node 20+
- Rust 1.70+
- Hardhat (`cd contracts && npm install`)

```bash
make build-lib-macos        # or build-lib-linux / build-lib-windows

npm install

```

## Commit standards

### DCO sign-off (required)

Every commit must include a Sign-off line. This certifies you have the
right to contribute the code under Apache 2.0 — see the
[Developer Certificate of Origin](https://developercertificate.org/).

```bash
git commit -s -m "your message"
# adds: Signed-off-by: Your Name <your@email.com>
```

PRs without DCO will be auto-blocked by CI.

### Commit message format

We prefer (but do not enforce) Conventional Commits:

```
<type>(<scope>): <subject>

<body>

Signed-off-by: ...
```

Types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `security`.

## Pull request process

1. **Open an issue first** for non-trivial changes (>50 lines).
2. **Branch from `master`**: `git checkout -b feat/your-feature`.
3. **Add tests** for new behavior.
4. **Run all tests locally** before pushing.
5. **Sign your commits**: `git commit -s`.
6. **Open a PR** linking to the issue.
7. **Wait for review**: we aim to triage within 7 days. Large PRs may
   take longer. Pings are welcome after 14 days of silence.
8. **Address feedback**: push fixes; do not force-push during review.
9. **Merge**: we squash-merge after approval.

## Code style

- **Go**: `gofmt`, `go vet`. (We don't currently mandate a stricter
  linter, but consistent style is appreciated.)
- **TypeScript**: project ESLint config.
- **Rust**: `cargo fmt`, `cargo clippy`.

CI enforces the stricter rules. Run locally before pushing.

## Reporting security issues

**Do NOT open GitHub issues for security bugs.** Email
`security@arcsign.io`. See [`SECURITY.md`](SECURITY.md) for the full
disclosure policy, PGP key, and Hall of Fame.

## Trademark

When discussing ArcSign in your contributions, you may freely reference
the name. **Do not** use the ArcSign name or logo for forks distributed
under that name. See [`TRADEMARK.md`](TRADEMARK.md).

## Release checklist (maintainers)

Every tagged release updates documentation **before** the tag. Run this list
in order:

1. **CHANGELOG.md** — add a `## [vX.Y.Z] — YYYY-MM-DD — <title>` section. The
   Release workflow extracts the release note from this heading, so it must
   exist before the tag.
2. **ROADMAP.md** — move newly shipped items into "Recently Shipped"; bump
   "Last reviewed".
3. **Cross-file audit** — grep for drifting facts across `README.md`,
   `ROADMAP.md`, and `CLAUDE.md` (chain counts, feature counts, provider
   matrix). Fix any mismatch (e.g. "6 vs 7 EVM chains").
4. **Version bump** — `dashboard/package.json`,
   `dashboard/src-tauri/tauri.conf.json`, `dashboard/src-tauri/Cargo.toml`,
   and the top-level `version` in `dashboard/package-lock.json`.
5. **Tag + push** — `git tag vX.Y.Z && git push arcsignio vX.Y.Z` to trigger
   the 3-platform Release workflow.
